## 🔧 핵심 구현 기술

### 1. Gameplay Ability System(GAS) 기반 확장형 전투 아키텍처
- 언리얼 엔진의 스킬 시스템 GAS를 전면 도입하여 기본 이동을 제외한 대시, 스킬 공격, 피격, 사망 처리를 모두 어빌리티(Ability) 단위로 모듈화하고 확장성 있는 전투 구조를 설계했습니다.
- GameplayTag를 적극 활용하여 캐릭터의 상태 판정, 스킬 분기, 상호작용 조건 등을 데이터 기반으로 통제함으로써 복잡한 전투 로직의 결합도를 낮추었습니다.

### 2. 로직과 연출의 분리 및 데이터 주도적 스탯 관리
- AttributeSet과 GameplayEffect를 연동하여 스킬에 따른 데미지 계산, 상태 이상 효과 적용 로직을 독립적인 데이터 자산으로 분리해 재사용성을 극대화했습니다.
- GameplayCue를 활용하여 스킬 발동 시점의 시각적 이펙트를 핵심 논리 코드와 완벽히 분리 제어함으로써 안전하고 깔끔한 코드 구조를 유지했습니다.

### 3. Behavior Tree 기반 체계적인 AI 시스템 로직 구성
- 언리얼 엔진의 Behavior Tree와 Blackboard를 활용하여 적 AI의 상황 인지, 스킬 선택, 추적, 공격 패턴을 유기적으로 연결하는 행동 트리 로직을 구축했습니다.
- 플레이어의 피격 및 사망 상태, 거리 진입 여부 등의 Blackboard 데이터를 기반으로 AI가 적절한 스킬 태스크를 수행하도록 설계하여 역동적인 싱글 플레이 전투 환경을 구현했습니다.

### 4. Listen Server 기반 멀티플레이어 동기화 및 서버 권한 검증
- Listen Server 환경을 고려하여 네트워크를 구성하고 클라이언트의 임의 변조를 방지하기 위해 발사체 스폰, 피격 판정, 데미지 처리 등의 핵심 판정 로직을 서버 권한 기반으로 안전하게 설계했습니다.
- 언리얼 네트워크 리플리케이션을 적용하여 분리된 클라이언트 환경에서도 캐릭터의 이동, 애니메이션, 피격 상태 등이 지연 없이 정확하게 동기화되도록 구현했습니다.
- SeamlessTravel을 이동하여 멀티플레이 진입 시 각 플레이어가 캐릭터 선택 창에서 고른 캐릭터 클래스 정보를 PlayerState에 유지한 채 레벨 이동을 하게 구현했습니다.
- 대전 게임 특성상 즉각적인 체감을 위해 이동/공격 어빌리티는 로컬에서 예측 실행되게, 피격/죽음 등의 핵심 판정 결과 이후 실행되는 어빌리티는 서버에서 우선 실행 후 로컬에 전파되게 설계했습니다.


## 🚀 트러블 슈팅

### 1. 멀티플레이 환경에서의 데이터 유지 및 개별 스폰 구조 설계
- **문제 상황:** - 싱글플레이에서 사용하던 GameInstance 방식은 각 클라이언트의 로컬 메모리에만 존재하므로 서버가 클라이언트의 캐릭터 선택 정보를 참조할 수 없는 구조적 한계가 있었습니다.
  - 일반적인 OpenLevel 사용 시 세션 연결이 끊기며 데이터가 휘발되는 문제와 모든 플레이어에게 서버의 기본값인 DefaultPawn이 할당되는 현상을 해결해야 했습니다.
- **해결 방안:** - 세션 유지가 가능한 Seamless Travel을 도입하고 레벨 전환 시에도 서버와 클라이언트 간에 데이터가 복제되는 PlayerState에 캐릭터 클래스 정보를 저장하여 데이터 유실을 방지했습니다.
  - GameMode의 GetDefaultPawnClassForController를 오버라이드하여 캐릭터 스폰 직전 각 플레이어의 PlayerState를 참조해 선택한 캐릭터 클래스를 동적으로 주입하도록 구현했습니다.

### 2. 커스텀 BTTask를 통한 AI 이동 한계 해결
- **문제 상황:** - 엔진 기본 MoveTo와 NavMesh 시스템은 지면의 연결성에 의존하기 때문에 캐릭터가 점프하거나 건물 옥상으로 이동하는 등 수직적 위치 차이가 발생하는 상황에서 AI가 경로를 찾지 못하고 멈추는 문제가 있었습니다.
- **해결 방안:** - NavMesh의 데이터 연결 여부와 관계없이 타겟과의 상대적 좌표(거리 및 높이)를 실시간으로 계산하는 커스텀 BTTask(CustomMove)를 개발했습니다.
  - 단순히 걷는 이동에 그치지 않고 GAS와 연동하여 AI가 상황에 맞춰 Dash, Jump, Double Jump 어빌리티를 발동시켜 자연스럽게 이동하도록 설계했습니다.
- **한계점:** - 현재 로직은 타겟과의 수평/수직 거리를 기준으로 한 단순 조건문 방식이기에 복잡한 지형에서 기존 언리얼의 MoveTo처럼 최적의 경로를 계산하여 이동한다는 느낌보다는 단조로운 추적 패턴을 보인다는 한계가 있습니다. 향후 환경 쿼리 시스템(EQS)이나 고도화된 알고리즘을 도입하여 더욱 지능적인 기동을 구현할 과제가 남아있습니다.
